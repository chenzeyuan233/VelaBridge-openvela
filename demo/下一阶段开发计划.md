# 下一阶段开发计划

## 一、当前 MVP 已完成内容

根据当前仓库内容，VelaBridge 已完成第一阶段 MVP 的项目初始化与演示基础材料：

- 已建立基础仓库结构，包括 `docs/`、`hardware/`、`firmware/`、`ai_agent/`、`demo/`、`logs/` 和 `ai_coding_logs/`。
- 已完成项目首页 README，明确项目名称、简介、目标用户、核心功能、三天 MVP 目标、目录结构、当前开发状态和 Apache-2.0 开源协议。
- 已完成 `docs/项目说明.md`，说明项目概述、应用场景、目标用户、创新点、第一阶段 MVP、第二阶段增强功能、安全隐私边界和比赛展示思路。
- 已完成 `hardware/BOM.md`，列出 openvela 开发板、屏幕、麦克风、喇叭/功放、实体按键、震动马达、电池、面包板和可选传感器等硬件清单。
- 已完成 `firmware/README.md`，定义 openvela 端第一阶段需要验证的屏幕、按键、震动、语音播报和日志保存交互。
- 已完成 `ai_agent/velabridge_ai_demo.py`，提供无第三方依赖的 Python MVP 脚本，演示字幕简化、快捷回复推荐、沟通摘要和设备日志生成。
- 已完成 `ai_agent/prompt_examples.md`，整理字幕简化、快捷回复推荐和沟通摘要生成的提示词示例。
- 已完成 `logs/demo_log_sample.json`，提供字幕显示、快捷回复、盲人模式、震动提醒、紧急求助和 AI 摘要生成的设备日志样例。
- 已完成 `ai_coding_logs/day1_gpt_codex_log.md`，记录 ChatGPT / Codex 参与项目方向确定、仓库结构设计、README、三天 MVP 计划和 AI Demo 脚本生成。

当前阶段仍是 MVP 原型准备与基础能力验证阶段，尚未完成真实 openvela 硬件联调、真实语音识别接入和可穿戴外观封装。

## 二、接下来 3 天开发任务

### 队员 A：项目管理、文档、比赛材料、视频脚本

- 维护每日任务看板，确认四名队员的当天目标、阻塞问题和验收标准。
- 根据硬件和 AI 联调进展更新 README、项目说明、三天 MVP 任务清单和本开发计划。
- 整理比赛提交材料，包括项目简介、技术路线、创新点、用户场景、安全隐私边界和后续规划。
- 编写 1–2 分钟演示视频脚本，明确镜头、旁白、设备动作和展示顺序。
- 每天整理 AI Coding 日志，记录 ChatGPT / Codex 的使用方式、生成内容、修改内容和人工确认结果。

### 队员 B：openvela 环境、屏幕、按键、震动、语音播报

- 跑通 openvela 开发环境，确认编译、烧录、串口日志和基础示例可运行。
- 完成 OLED/LCD 屏幕显示测试，至少能显示固定字幕、模式名称和求助提示。
- 完成实体按键 A/B/C 的短按、长按识别，输出可读调试日志。
- 完成震动马达控制，支持短震 1 次、短震 2 次、长震 1 次和连续强震。
- 完成喇叭/功放播报测试，确认快捷回复和求助语音可以稳定播放。
- 打通基础事件日志保存流程，记录按键、字幕、模式切换、震动和求助事件。

### 队员 C：AI Agent、字幕简化、快捷回复、摘要生成

- 优化 `ai_agent/velabridge_ai_demo.py` 的字幕简化规则，让输出更短、更适合小屏幕阅读。
- 扩展快捷回复推荐场景，覆盖课堂、医院、窗口服务、问路、紧急求助和嘈杂环境。
- 设计盲人模式语音菜单文案，包括进入提示、按键说明、返回提示和求助确认。
- 优化沟通摘要生成逻辑，确保摘要只基于日志中已有事件，不额外推断敏感信息。
- 统一设备日志字段，保持 Python Demo、JSON 样例和 openvela 端日志格式一致。
- 准备后续接入 GPT API 或本地模型的接口草案，但 MVP 演示仍保留无外部依赖版本作为稳定兜底。

### 队员 D：硬件接线、测试、演示场景、外观原型

- 根据 BOM 准备 openvela 开发板、屏幕、按键、震动马达、喇叭/功放、电池、杜邦线和面包板。
- 完成硬件接线图草稿，标注 GPIO、供电、地线、音频输出和震动马达控制接口。
- 配合队员 B 做分模块测试，记录每个模块的接线方式、测试结果和异常现象。
- 设计三类演示场景：听障字幕沟通、视障盲人模式、紧急求助。
- 制作可穿戴外观原型，可使用纸壳、3D 打印件、绑带或简易外壳展示设备佩戴方式。
- 准备演示现场的备用线材、备用电源、备用按键和故障替代方案。

## 三、硬件端任务清单

- 屏幕显示测试：显示固定字幕、AI 简化字幕、当前模式、按键提示和求助状态。
- 按键输入测试：识别按键 A、按键 B、按键 C 的短按事件，并输出串口日志。
- 震动马达测试：验证短震 1 次、短震 2 次、长震 1 次和连续强震的可感知差异。
- 喇叭播报测试：播放“请说慢一点，谢谢”“我需要帮助”和盲人模式菜单提示。
- 日志保存测试：保存字幕显示、快捷回复、模式切换、震动提醒和紧急求助事件。
- 长按/短按交互测试：确认短按不误触长按，长按按键 C 可进入盲人模式，长按求助键可触发紧急求助。

## 四、AI 端任务清单

- 字幕简化：将较长口语句子压缩为适合小屏幕阅读的短句，优先保留时间、地点、动作和请求。
- 快捷回复推荐：根据课堂、医院、窗口服务、问路、求助等场景推荐礼貌、简短、可直接播报的回复。
- 盲人模式语音菜单：生成“当前为盲人模式”“短按 A 播报常用回复”“长按求助键触发求助”等语音提示文案。
- 沟通摘要：根据设备事件日志生成简洁摘要，避免保存原始音频和不必要个人隐私。
- 设备日志格式：统一 `timestamp`、`event_type`、`content`、`mode` 字段，便于 openvela 端和 AI Demo 对接。
- 后续接入计划：第二阶段可接入 GPT API 做更自然的摘要和回复推荐，也可评估本地小模型，保证隐私模式和离线兜底能力。

## 五、演示视频脚本草案

### 0:00–0:15 项目介绍

画面：队员佩戴或手持 VelaBridge MVP 原型，旁边展示 openvela 开发板、屏幕、按键、震动马达和喇叭模块。

旁白：VelaBridge 是一个基于 openvela 的多模态无障碍 AI 沟通与伴行终端，面向听障、视障和语言表达困难人群。当前展示的是三天内完成的 MVP 原型。

### 0:15–0:40 听障场景：字幕显示 + 快捷回复

画面：一名队员模拟窗口服务人员说话，设备屏幕显示简化字幕“请说慢一点，我需要看字幕。”。

操作：用户短按按键 A，设备显示并播报“请说慢一点，谢谢”。

旁白：在听障沟通场景中，设备把外界语音转换为更易阅读的字幕，并通过按键触发常用回复，降低即时沟通压力。

### 0:40–1:05 视障场景：盲人模式 + 语音播报 + 震动提醒

画面：用户长按按键 C，设备进入盲人模式。

操作：喇叭播报盲人模式菜单，震动马达短震 2 次确认模式切换。

旁白：在视障场景中，用户不需要看屏幕，可以通过语音菜单、触感按键和震动反馈完成基础操作。

### 1:05–1:25 紧急求助场景

画面：用户长按求助键，设备连续震动并播报“我需要帮助”。

操作：屏幕显示紧急求助状态，日志记录 `emergency_help` 事件。

旁白：当用户遇到紧急情况时，可以通过长按触发求助播报，同时设备记录事件日志，为后续联系人通知和定位扩展保留接口。

### 1:25–1:50 AI 摘要展示

画面：电脑终端运行 `python ai_agent/velabridge_ai_demo.py`，展示字幕简化、推荐回复、事件日志和沟通摘要。

旁白：AI Agent 根据设备日志生成沟通摘要。MVP 阶段默认不保存原始音频，只保存用户确认后的文字摘要和必要事件记录。

### 1:50–2:00 结尾

画面：展示设备原型和项目仓库页面。

旁白：下一阶段，我们将继续接入真实 openvela 硬件链路、语音识别、更多传感器和隐私保护能力，让 VelaBridge 从可演示原型逐步迭代为更完整的无障碍沟通设备。

## 六、风险与解决方案

| 风险 | 影响 | 解决方案 |
| -- | -- | -- |
| openvela 环境搭建失败 | 无法完成真实开发板联调，影响硬件演示 | 提前准备官方示例、串口日志和环境截图；保留 PC 模拟脚本作为兜底；安排队员 B 优先解决环境问题 |
| 硬件模块到货慢 | 屏幕、喇叭、震动或按键链路无法完整展示 | 优先采购本地可买模块；使用面包板和现有模块替代；演示时用串口日志或屏幕模拟未到货模块 |
| 语音识别短期跑不通 | 实时字幕无法从真实麦克风输入生成 | MVP 阶段使用预设文本或串口输入模拟语音识别结果；第二阶段再接入真实语音识别服务或本地模型 |
| 演示不稳定 | 视频录制或现场展示中断，影响比赛观感 | 每个功能录制独立备用片段；准备手动触发脚本；保留按键、震动、屏幕和 AI 摘要的分模块演示 |
| 文档和 AI Coding 日志遗漏 | 比赛材料不完整，难以体现 AI-Native 开发过程 | 队员 A 每天结束前统一检查 README、项目说明、开发计划和 AI Coding 日志；每次新增代码或文档都同步记录 |
